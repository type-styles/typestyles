Add `variantStrategy: 'attribute'` to `styles.component()`'s dimensioned config — each `variants` option compiles to a `&[data-{dimension}="{option}"]` selector scoped under the single `base` class instead of a discrete class, and the callable returns a `{ className, attrs, props }` result (string-coercible via `toString()`/`Symbol.toPrimitive`, so `String(...)`/`cx(...)` still work) instead of a class string. Boolean dimensions (`{ true, false }` option keys) are presence-based: `true` → `&[data-x]`, `false` → `&:not([data-x])`. `compoundVariants` compile to a single combined attribute selector with no compound class or runtime matching. Set `defaultVariantStrategy: 'attribute'` on `createStyles`/`createTypeStyles` to make this the default across every component from that instance, overridable per-component. This lets design systems that mix classnames and `data-*` attributes for styling (e.g. `<button class="btn" data-variant="primary" data-size="small">`, matching Radix/shadcn conventions) author that pattern with the same typed `variants`/`compoundVariants`/`defaultVariants` shape used for class-based components. Only the plain dimensioned config shape supports it — not `slots` or flat configs. See `specs/attribute-driven-variants.md`.
